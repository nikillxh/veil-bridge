# Architecture

The bridge is built from three independent pillars connected by two distinct
zero-knowledge proofs. Keeping the two proofs separate is deliberate: one
removes trust in the relayer, the other provides privacy.

## Two proofs, two jobs

| Proof | Generated by | Where | Proves |
|-------|--------------|-------|--------|
| **Inclusion proof** | Rust relayer (SP1, server-side) | source block | A `ShieldedVault` commitment-tree root was in source state at block N |
| **Shielded-claim proof** | Claimer (Circom Groth16, client-side) | QIE | Knowledge of a commitment leaf under an accepted root + a fresh nullifier |

## Pillar 1 - Smart contracts

### Source chain (`contracts-source/`)
- `MerkleTreeWithHistory.sol` - fixed-depth (20) incremental Poseidon Merkle
  tree over BN254, with a rolling 30-root history and a fixed-slot `latestRoot`
  mirror (slot 3) so the relayer can prove a single, stable storage slot.
- `ShieldedVault.sol` - locks a fixed `denomination` of an asset and inserts the
  caller's `commitment` (no destination address is recorded), emitting `Deposit`.
- `IHasher.sol` - Poseidon abstraction. Deployments use the `circomlibjs`
  Poseidon bytecode (identical to the circuit); the unit tests use a
  keccak-based test hasher.

### QIE (`contracts-qie/`)
- `BridgeUpdater.sol` - the light client. `updateRoot(publicValues, proof)`
  calls the SP1 verifier; only on success does it record the proven root.
- `ShieldedPool.sol` - `withdraw(...)` verifies a Groth16 membership proof
  against an accepted root, enforces nullifier uniqueness, then mints. Supports
  a `relayer`/`fee` for gasless claims from a brand-new wallet.
- `WrappedToken.sol` - ERC20 mintable only by the pool.
- `verifiers/WithdrawVerifier.sol` - snarkjs-exported Groth16 verifier.
- `libraries/BridgePublicValues.sol` - the ABI schema shared verbatim with the
  Rust guest, preventing public-value drift.

## Pillar 2 - SP1 light client (`sp1-program/`, `relayer/`)

- `bridge-core` - pure verification logic used by **both** the guest and the
  relayer: decode the block header (recompute its hash), verify the
  Merkle-Patricia account proof against `header.stateRoot`, then verify the
  storage proof for the `latestRoot` slot against the account's storage root.
  Output: `(blockHash, blockNumber, vault, root)`, ABI-encoded as public values.
- `program` - the SP1 guest: reads the witness, runs `verify_inclusion`,
  commits the public values.
- `relayer` - watches the source vault, builds the witness via `eth_getProof`
  + `eth_getBlockByNumber` (at the `finalized` block), proves it, and submits to
  `BridgeUpdater`. The native verification mode runs the same inclusion logic
  in-process for local integration testing; `Sp1Prover` (`--features sp1`)
  produces a Groth16-wrapped proof verified on chain.

## Pillar 3 - Privacy layer (`circuits/`, `client/`)

- `withdraw.circom` - proves `commitment = Poseidon(nullifier, secret)` is a
  leaf under the public `root`, exposes `nullifierHash = Poseidon(nullifier)`,
  and binds `recipient/relayer/fee/refund` so a relayer cannot redirect funds.
- `client` - generates notes, mirrors the on-chain tree to compute Merkle
  paths, produces the Groth16 proof with snarkjs, and submits deposit/claim.

## Data flow

1. Depositor creates a note `(nullifier, secret)`, computes
   `commitment = Poseidon(nullifier, secret)`, and calls `vault.deposit`.
2. Relayer detects the new root, proves its inclusion at a finalized block,
   submits to `BridgeUpdater`, which verifies and records the root.
3. From a fresh wallet, the claimer rebuilds the tree from `Deposit` events,
   generates a membership proof for their note, and calls `pool.withdraw`.
4. The pool verifies the proof, checks the nullifier is unused, and mints the
   wrapped asset to the recipient. The source-to-recipient link is severed.
